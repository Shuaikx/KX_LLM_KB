## 一、总体判断

这组文档本质上是一套 **AOE3D 项目 UI 基础能力知识库**，覆盖四大方向：

1. **通用 UI 组件与复用规范**：控件库、ListView/ListBox、TemplateBox、筛选框、多 EditBox、功能说明弹窗等。
2. **统一交互与提示体系**：Tips、MsgBox、提示条、改名弹窗、触发式弹窗、StackUI 多实例策略。
3. **性能与工程化优化**：倒计时从 Lua 挪到 C#、基于服务端时间戳的回调机制、分页拉取组件。
4. **渲染 / 适配 / 疑难问题排查**：3D Canvas 透明异常、Mask 淡出贴图 bug、动态分辨率边界问题、Android 多窗口 16:9 适配。

如果你是做功能开发，这个目录最值得优先吸收的不是“单个接口”，而是背后的统一思路：

- **少裸写，多走通用组件和统一入口**。
- **少在 Lua OnTick/Update 做重复工作，能集中到 C# 或统一调度就集中**。
- **UI 弹窗、Tips、提示条要标准化，不要各写各的**。
- **遇到 UI 显示异常时，先从渲染顺序、Alpha、坐标空间、Canvas 适配链路排查**。

## 二、按主题分类总结

### 1. 通用 UI 组件 / 编辑器工具

#### 1) UI控件库规范与说明
- 目标：把 button、toggle、slider 等基础控件做成模板 prefab，统一风格并支持一键 apply。
- 使用方式有两类：
  - 基于 Excel 批量生成菜单项 / Temp 代码。
  - 基于编辑器窗口直接选中节点后放置 UI 模板。
- 价值：**降低 UI 搭建成本、统一样式、减少重复造轮子**。
- 关键路径：`AOE3DUIEditor\Assets\UIProjTools\UITools\Editor\UITempPopMenu\...`

#### 2) lua中常用的UI容器组件
- 总结了三类常见容器：
  - `ListBox / UIListView / UIListViewEasy`
  - `TemplateBox / UITemplateBox`
  - 裸写方式
- 推荐倾向：
  - 简单场景优先 `UIListViewEasy` / `UITemplateBox`
  - 复杂、可复用 item 用 `UIListView`
  - 裸写灵活但维护成本高
- 结论：**这篇是 UI 列表/模板复用的入门基石文档**。

#### 3) ListView分页拉取组件
- 组件拆分为 `SplitPageController / SplitPageData / SplitPageView`。
- 通过传入 `startIdx/endIdx` 按需拉取当前视野范围的数据，解决超大网络包问题。
- 典型场景：成员列表、部队列表等数据量大且只需局部显示的界面。
- 结论：**这是 UI 与网络分页协同的标准做法**。

#### 4) 筛选框使用说明
- 统一筛选框分为单选、多选两类。
- 多选方案围绕 `PanelFileterBaseData` 初始化数据，再用 `UIUtil.CreateCommonFilterBtn` 注册按钮和刷新表现。
- 重点不在“怎么画 UI”，而在**把筛选条件、排序条件、按钮样式绑定成统一数据驱动流程**。

#### 5) 多EditBox检查组件
- 提供 `UIFormatView` 统一管理多个 EditBox 的输入校验。
- 支持：特殊字符、繁体过滤、长度检查、异步敏感词检查。
- 适合多输入框场景统一管控输入合法性和按钮状态。
- 结论：**避免每个 EditBox 都各自写一遍校验逻辑**。

#### 6) Text组件打字机效果使用说明
- Text 组件内建打字机效果，支持：
  - 从文本或 key 启动
  - 滑动 / 非滑动两种显示方式
  - 跳过、停止、重启
  - 富文本长度计算和富文本截断
- 适合剧情、引导、叙事文本。

#### 7) ListBox元素聚拢动画
- `PlayGatherAnim(...)` 支持在列表中抽走一个元素后，其余元素自动聚拢。
- 适合卡牌抽离、列表收缩等视觉反馈场景。

  - 开发同学容易混用。
  - 目标是提供统一接口，通过参数区分具体显示版本。
- 结论：**统一入口 > 各种零散提示 API**。

#### 12) 物品通用Tips小弹窗
- 定义了三种常见物品弹窗：
  - 道具弹窗 `PopupBox_HoverInfo_Item`
  - 宝箱弹窗 `PopupBox_BoxInfo`
  - 通用物品详情弹窗（自动判断显示哪种）
- 适合 item/tips 相关功能快速接入。

#### 13) 通用改名弹窗查询文档
- 统一更名输入框 + 敏感词检查 + 长度校验 + 消耗展示 + 二次确认弹窗。
- 通过 `PopupBox_CommonChangeNameParamData` 实现全量定制，也提供快捷调用。
- 支持：货币 / 资源 / 道具消耗，服务端基于 `checkInputType` 做敏感词校验。
- 结论：**涉及更名、重命名、输入确认的功能建议直接走这套**。

#### 14) 通用功能说明弹窗
- 支持图 / 动画 / 视频多页切换展示。
- 复用配表 `T_图文引导`，程序侧只需传功能 ID。
- 典型用途：功能说明、首次教学、说明页轮播。

#### 15) 触发式弹窗统一管理规则
- 文档很短，核心接口：`mgr.ui:ShowStackUIOnSceneEmpty()`。
- 说明了 UI 打开的时机可能来自玩家输入、网络回包、协程条件满足等不同生命周期。
- 重点是：**通过统一管理触发式弹窗的打开优先级，避免时机冲突**。

#### 16) 新增MultiInStack Flag类型
- 解决同一个 StackUI 页面被“再次打开”时需要保留原状态的问题。
- `MultiInStack` 允许栈中存在多个同名 prefab 实例，而不是复用同一个。
- 提供回调：
  - `OnMultiModeSameUIShow`
  - `OnMultiModeRestore`
- 适合：A 页面内跳转打开“同名但另一份状态”的页面，关闭后希望恢复原页面现场。

### 3. 时间、性能与统一调度

#### 17) 基于C#的倒计时使用说明文档
- 明确指出问题：Lua 里大量 `OnTick` / `OnUpdate` 做倒计时文本拼接，性能差、风格不统一。
- 方案：把倒计时字符串拼接放到 C#，通过统一接口注册 / 反注册。
- 支持 4 种默认格式、前后缀拼接、剩余时间分段变色。
- 结论：**倒计时优先用统一注册，不要在 Lua 每秒手写 setText**。

#### 18) 指定时间戳回调机制
- 目标：替代基于本地时间轮 `scheduler` 的不稳定回调，改用服务端时间戳调度。
- 适用场景：
  - 和倒计时配套
  - 需要在准确服务端时间点触发回调
  - 避免 pause / GM 改时间引起的误差
- 设计：按时间戳聚合 observer，始终取最近时间点作为回调节点。
- 结论：**凡是依赖服务端时间语义的延迟触发，优先用这套而不是本地 scheduler**。

### 4. UI 渲染 / 适配 / 疑难问题

#### 19) 3D Canvas下透明UI显示异常问题
- 问题：3D Canvas 下 Mask 更新了 RT 的 Alpha，导致与背景混合后出现“透贴”。
- 根因：3D Canvas 与 2D Canvas 的渲染顺序不同。
- 修复：让 `Pic_Mask` 不更新 Alpha 通道（调整 Color Mask）。
- 结论：**这是典型的 RT Alpha / Blend 链路问题**。

#### 20) UGUI动态设置mask显示异常问题
- 需求：动态设置 mask 的 FadeTexture。
- Bug 根因：从 none 切到有 FadeTexture 时，没有更新存储在顶点数据里的 fadeIndex，导致采样到的 alpha 为 0，看起来“没渲染”，本质上是全透明。
- 修复结论：**设置顶点脏标记，强制刷新顶点数据**。
- 这篇文档的价值很高，因为它把 RenderDoc / frame debug / 顶点数据排查链路讲清楚了。

#### 21) UI动态适配不同分辨率超出边界问题
- 问题：不同分辨率下，弹出的属性界面被遮挡，根因是混用了像素空间坐标与 canvas 变化前空间坐标进行边界判断。
- 修复思路：把运算统一到 **rect 局部空间** 比较，而不是混合不同坐标空间。
- 结论：**涉及动态适配和边界判断时，先统一坐标空间再谈逻辑**。

#### 22) Android 多窗口 / 小窗 16:9 适配说明（杂交华为版本专项适配）
- 目标：在分屏 / 小窗 / 自由窗口中，让 UnityPlayer 在任务框内按 16:9 横屏居中显示，四周留黑边。
- 当前方案：
  - Window / Decor 保持 `MATCH_PARENT`
  - 真正约束的是 `UnityPlayer` 视图尺寸
  - 在多窗口生命周期节点反复重申布局
- 特别说明了华为等 ROM 对窗口固定尺寸修改的限制，因此不再依赖直接改 Window 尺寸。
- 结论：**内容层 letterbox 比窗口层硬改尺寸更稳**。

- 从标题看是 Editor 环境下 UI 加载失败问题的专项排查记录，但可直接复用的信息不多。

## 三、这批文档反映出的共性规范

### 1. 能统一就统一
- UI 组件尽量走模板库 / 通用组件。
- Tips / MsgBox / 提示条尽量走统一入口，不要复用历史散装接口。
- 输入校验、倒计时、分页拉取等都在努力从“业务散写”收敛到“底层复用”。

### 2. 能数据驱动就不要硬编码
- 控件库通过 Excel / 编辑器工具驱动。
- 功能说明弹窗靠配表一页页配置。
- 筛选框、列表、消息格式等都在强调数据结构先行。

### 3. Lua 只保留业务逻辑，重复重活往底层沉
- 倒计时文本拼接放到 C#。
- 时间点触发交给服务端时间戳机制。
- 多 EditBox 校验、多列表分页等用统一组件做。

### 4. UI 问题排查优先看“空间”和“渲染链路”
- 是不是 Alpha 被改坏了？
- 是不是 RT / Mask / Blend 顺序导致的？
- 是不是用了两种不同坐标空间？
- 是不是顶点数据没刷新？

这几篇问题定位文档基本都在反复证明这一点。
