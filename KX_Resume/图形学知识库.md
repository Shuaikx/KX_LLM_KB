# 渲染管线 / Rasterization 光栅化管线

## 渲染管线结构

渲染管线的基本结构：应用阶段，几何阶段，光栅化阶段，像素着色阶段

光栅化渲染管线（Rasterization Pipeline）可以概括为以下 6 个核心步骤：

---

**Step 1 — Transform / Position objects in the world（场景变换）**

将模型从局部空间依次变换到世界空间，再变换到相机空间，最后投影到裁剪空间。

- **Model Matrix**：模型空间 → 世界空间（位移、旋转、缩放）
- **View Matrix**：世界空间 → 相机/观察空间
- **Projection Matrix**：相机空间 → 裁剪空间（NDC）

> 对应 GPU 管线中的 **Vertex Shader**，逐顶点执行，有多少顶点就执行多少次。 

---

**Step 2 — Project objects onto the screen（投影到屏幕）**

将裁剪空间坐标做**透视除法**，得到 NDC（Normalized Device Coordinates），再映射到屏幕像素坐标（Viewport Transform）。同时进行**视锥体裁剪**，丢弃视野之外的图元。

> 这一步是纯固定硬件操作，不可编程。

---

**Step 3 — Sample triangle coverage（三角形覆盖采样 / 光栅化）**

将屏幕上的三角形离散化——遍历三角形包围盒内的所有采样点（像素中心或 MSAA 子采样点），判断每个点是否在三角形内部。

- 使用**重心坐标（Barycentric Coordinates）**或边函数（Edge Function）判断覆盖关系。
- 被覆盖的采样点成为候选 **Fragment（片元）**。

> 光栅化是 CPU → GPU、几何 → 像素的关键跨越。

---

**Step 4 — Interpolate triangle attributes at covered samples（属性插值）**

对每个被覆盖的片元，用重心坐标对三角形三个顶点的属性进行**透视正确插值（Perspective-Correct Interpolation）**：

- UV 坐标（纹理坐标）
- 顶点颜色
- 法线向量（Normal）
- 切线向量（Tangent）
- 自定义 varying 变量

> 插值后的属性作为 **Fragment Shader 的输入**。

---

**Step 5 — Sample texture maps / Evaluate shaders（纹理采样 / 执行着色器）**

对每个片元执行 **Fragment Shader（片元着色器）**，完成：

- **纹理采样**：根据插值后的 UV 坐标从纹理图集中取颜色（可能经过 Mipmap 选级、双线性/三线性过滤）。
- **光照计算**：应用光照模型（Blinn-Phong / PBR）计算最终颜色。
- **其他效果**：法线贴图扰动、自发光、透明度计算等。

> Fragment Shader 是完全可编程的，执行次数 = 被覆盖的有效片元数量（与屏幕分辨率/模型大小相关，与顶点数无关）。

---

**Step 6 — Combine samples into final image（合成最终图像）**

将片元着色结果写入 Framebuffer，期间经过一系列**逐片元测试**：

- **Depth Test（深度测试 / Z-Test）**：比较当前片元深度与深度缓冲（Z-Buffer）中已有值，保留最近的片元，实现遮挡关系。
- **Stencil Test（模板测试）**：自定义遮罩，用于描边、镜像反射等效果。
- **Alpha Blending（透明混合）**：将半透明片元与已有颜色混合（`src_alpha × src_color + (1 - src_alpha) × dst_color`）。

最终所有采样点的颜色合并后输出到屏幕 / 后处理 Pass。

---

**管线全貌（总结）：**

```
顶点数据
  │
  ▼
[Step 1] Vertex Shader（顶点变换：Model → World → View → Clip）
  │
  ▼
[Step 2] 投影 + 裁剪 + Viewport 变换（→ 屏幕空间）
  │
  ▼
[Step 3] 光栅化：三角形 → 片元（Sample Coverage）
  │
  ▼
[Step 4] 属性插值（UV、Normal、Color 等）
  │
  ▼
[Step 5] Fragment Shader（纹理采样 + 光照计算）
  │
  ▼
[Step 6] 逐片元测试（Depth / Stencil / Blend）→ Framebuffer → 屏幕
```

## 光照模型 / Blinn-Phong

**标准回答：**

常见光照模型：Lambert（漫反射）、Phong、Blinn-Phong、PBR（基于物理）。

**Blinn-Phong 模型由三部分组成：**

1. **环境光（Ambient）**：`Ka × Ia`，模拟全局间接光，防止阴影完全黑暗。
2. **漫反射（Diffuse）**：`Kd × Id × max(0, N·L)`，与法线和光源方向夹角有关，模拟粗糙表面散射。
3. **镜面反射（Specular）**：`Ks × Is × max(0, N·H)^shininess`，H 是半程向量（L+V 归一化）。

**与 Phong 的区别：** Phong 用反射向量 R 与视线 V 的夹角，Blinn-Phong 改用半程向量 H，计算更高效且在某些情况下结果更自然。

## Shadow Map

**shadow map 原理**

1. **第一步（光源视角渲染）**：从光源位置渲染场景，将每个像素的深度值记录到深度纹理（Shadow Map）中。
2. **第二步（相机视角渲染）**：正常渲染场景，对每个片元：
   - 将其世界坐标变换到光源裁剪空间。
   - 从 Shadow Map 采样该位置的深度值（光源能看到的最近深度）。
   - 若当前片元深度 > Shadow Map 中的值（加偏移 bias），则该点在阴影中。

**常见问题：**

- **Shadow Acne（自阴影瑕疵）**：用 depth bias 解决。
- **锯齿（Aliasing）**：用 PCF（Percentage Closer Filtering）软化阴影边缘。

# Geometry 几何

### 2000 个顶点的模型，VS 和 FS 各执行多少次？

**标准回答：**

- **Vertex Shader**：逐顶点执行，执行 **2000 次**。
- **Fragment Shader**：逐片元（屏幕像素）执行，次数取决于模型在屏幕上占据的**像素数量**，与顶点数无直接关系。若该模型在屏幕上覆盖 10 万像素，则 FS 执行 10 万次。

### 几何 Mesh

1. 什么是好的mesh
   1. 三角形mesh：顶点的degree为6；
      三角形尽量接近正三角形，避免出现狭长的针状三角形（符合Delaunay，三角形的外接圆范围内不包含任何其他顶点），这样可以使得插值比较的丝滑。
      例外是如圆柱体的侧面可以使用细长的三角形进行表示。
   2. 四边形mesh：顶点的degree为4
      奇异点是degree不为4的点，只能保证切线平滑，但是曲率可能会跳变，也就是导致不规则的surface normal，或锯齿状的反射/散射
      使用catmull-clark细分，可以在第一次细分时就得到所有的奇异点，在后续的细分中不会产生新的奇异点。而奇异点的数量固定，可以使得算法的收敛速度非常一致

# Animation 动画

## Numerical Integration 数值积分

在计算机动画的物理模拟（物理动画）中，**数值积分（Numerical Integration）**的作用是：已知物体当前的状态和受力情况，计算它在下一时刻（下一帧）的位置和速度。

由于计算机无法像数学家那样求出解析解，它必须通过“小步快跑”（时间步长 $\Delta t$）的方式来逼近真实的运动轨迹。

------

### 1. 前向欧拉法 (Forward Euler / Explicit Euler)

前向欧拉是**显式（Explicit）**方法。它的逻辑最简单：**利用“当前时刻”的状态来推测“未来时刻”的状态。**

- **数学表达式：**

  1. $v_{n+1} = v_n + a_n \Delta t$（新速度 = 旧速度 + 当前加速度 × 时间步长）
  2. $x_{n+1} = x_n + v_n \Delta t$（新位置 = 旧位置 + 当前速度 × 时间步长）

- **直观理解：**

  就像开车时蒙住眼，每隔一秒睁眼看一次。你根据这一秒看到的车速和方向，假设在下一秒内保持不变，然后直接开过去。

- **优缺点：**

  - **优点：** 计算极快。只需要做加法和乘法，没有复杂的方程求解。
  - **缺点：** **极其不稳定**。对于像弹簧这样往复运动的系统，前向欧拉会不断增加系统的能量。
  - **现象：** 你的布料会突然炸开，或者原本应该绕圈运行的行星轨迹越绕越大，最后飞出屏幕。

------

### 2. 后向欧拉法 (Backward Euler / Implicit Euler)

后向欧拉是**隐式（Implicit）**方法。它的逻辑比较“烧脑”：**它要求“下一时刻”的状态必须符合“下一时刻”受力规律。**

- **数学表达式：**

  1. $v_{n+1} = v_n + a_{n+1} \Delta t$
  2. $x_{n+1} = x_n + v_{n+1} \Delta t$

  - *注意：等号右边出现了 $a_{n+1}$ 和 $v_{n+1}$，这意味着你在计算这一帧时，必须提前知道下一帧的状态。*

- **如何求解？**

  因为下一帧的状态是未知的，所以这变成了一个**求根问题**或**解方程组问题**。在动画引擎中，通常需要使用**牛顿迭代法（Newton's Method）**或解一个巨大的线性方程组。

- **优缺点：**

  - **优点：** **无条件稳定**。无论你的 $\Delta t$ 有多大，系统永远不会爆炸。它具有“数值阻尼”，能自动吸收多余能量。
  - **缺点：** 1.  **计算量巨大**：每一帧都要解矩阵方程。
    2. **过度平滑**：它会让动画看起来有点“肉”，像是在粘稠的液体里运动，丢失了一些生动的细节。

------

### 3. 核心对比：为什么稳定与不稳定？

为了理解这种差异，我们可以用**弹簧受力**作为例子：

| **特性**           | **前向欧拉 (Forward)**       | **后向欧拉 (Backward)**  |
| ------------------ | ---------------------------- | ------------------------ |
| **能量变化**       | 能量不断增加（数值不稳定性） | 能量不断衰减（数值阻尼） |
| **轨迹偏离**       | 总是切着圆弧向外飞           | 总是切着圆弧向内收缩     |
| **时间步长敏感度** | 必须非常小，否则会崩溃       | 可以很大，适合实时交互   |
| **实现难度**       | 一行代码搞定                 | 需要线性代数库解方程     |

------

### 4. 动画中的“折中方案”：半隐式欧拉 (Semi-implicit Euler)

在实际的游戏开发（如 **Unity** 或 **Unreal** 的物理引擎）中，最常用的其实是**半隐式欧拉法**（也叫 Symplectic Euler）：

$$v_{n+1} = v_n + a_n \Delta t$$

$$x_{n+1} = x_n + \mathbf{v_{n+1}} \Delta t$$

**区别在哪？** 在更新位置时，它使用的是**刚刚算出来的“新速度”**而不是“旧速度”。

这个小小的改变，使得它在保持计算简单的同时，**守恒性**大为提高，对于轨道运动和简单的物理反馈非常有效。

------

### 总结

- 如果你要做一个简单的粒子特效（火花、雨滴），**前向欧拉**或**半隐式欧拉**足够了。
- 如果你要模拟复杂的布料、头发或者是需要极高稳定性的物理系统，你必须使用**后向欧拉**（以及后续的线性方程求解器）。
