# 投递状态更新 & 面试经验记录

更新 `投递记录.md` 中某个岗位的状态，并可选记录面试经验（支持手动输入文字或导入音频文件自动总结）。

## 用法示例

```
/update-job                          # 不带参数，逐步引导
/update-job 2 已约面                 # 快速更新序号 2 的状态
/update-job 1 面试中 面试笔记.txt    # 更新状态并从文本文件读取面试记录
/update-job 2 面试中 recording.m4a  # 更新状态并从音频文件转录面试经验
```

## 执行步骤

### Step 1 — 确认目标岗位

1. 读取 `投递记录.md`，展示当前**投递列表表格**（仅序号、公司、岗位、状态列）供用户选择。
2. 询问用户：**要更新哪个序号的岗位？**（若命令参数已给出则跳过）

### Step 2 — 确认新状态

询问用户新状态（若参数已给出则跳过）：
```
📝 待投递 / ✉️ 已投递 / 📞 已约面 / ⏳ 面试中 / 🎉 已拿Offer / ❌ 已拒/放弃
```
同时询问**本次跟进备注**（可留空）。

### Step 3 — 处理面试经验（可选）

询问用户是否要记录面试经验，有三种输入方式：

**方式 A：不记录**
- 跳过此步骤。

**方式 B：手动输入文字**
- 用户直接粘贴或描述面试过程，Claude 将其整理为结构化面试小结（见格式规范）。

**方式 C：导入音频文件**
- 用户提供音频文件路径（.m4a / .mp3 / .wav / .aac 等）。
- 使用 Read 工具读取该音频文件，Claude 直接进行语音转录。
- 转录完成后，自动归纳为结构化面试小结（见格式规范）。
- 若 Read 工具无法读取该音频格式，提示用户安装 whisper 转录：
  ```bash
  pip install openai-whisper
  whisper recording.m4a --language zh --output_format txt
  ```
  然后请用户将生成的 `.txt` 路径再次提供（走方式 C 文本分支）。

### 面试小结格式规范

```markdown
#### 面试小结 — {日期}（{面试轮次，如一面/二面/HR面}）

- **面试形式：** 视频 / 现场 / 电话
- **时长：** 约 XX 分钟
- **面试官：** （若知道则填写，否则略）

**考察内容：**
- （按主题列点，例如：项目经历、算法题、系统设计、行为问题等）

**回答亮点：**
- （你认为回答得好的点）

**待改进：**
- （回答不流畅或没准备好的点）

**整体感受：** （1-2 句主观评价）
```

### Step 4 — 写入文件

对 `投递记录.md` 执行以下三处修改（用 Edit 工具精确替换，不要破坏其他内容）：

1. **投递列表表格**：找到对应序号的行，将状态列的 emoji 替换为新状态。

2. **对应岗位的跟进记录表**：在最后一行数据之后追加新行：
   ```
   | {今日日期} | {新状态文字描述，如"完成一面"} | {备注} |
   ```

3. **若有面试小结**：在该岗位的跟进记录表之后（`---` 分隔线之前）插入整理好的面试小结 Markdown。

### Step 5 — 知识点查漏补缺（若有面试经验输入）

**仅在用户提供了面试经验（方式 B 或 C）时执行此步骤。**

1. 读取 `客户端知识点.md` 的全部内容。
2. 逐条检查面试经验中出现的**技术考察点**（包括用户答得好的和答得不好的），判断该知识点是否已在 `客户端知识点.md` 中有对应章节：
   - **已覆盖**：跳过。
   - **未覆盖**：将其列为"待补充知识点"。
3. 对每个待补充知识点，在 `客户端知识点.md` 的**最合适章节**末尾追加完整的知识条目，格式与现有条目保持一致：

```markdown
### N. {知识点名称}

**标准回答：**

（系统性解释：定义、原理、对比、实际应用）

---
```

4. 若无新知识点需要补充，跳过文件修改，输出"客户端知识点.md 已完整覆盖本次考察内容"。

### Step 6 — 确认输出

输出一条简洁确认，包含：
- 岗位名称
- 更新后的状态
- 面试小结字数（若有）
- 新增到 `客户端知识点.md` 的知识点列表（若有），格式：`+ [章节] 知识点名称`

提示用户可运行 `/log-job` 继续录入新岗位。
