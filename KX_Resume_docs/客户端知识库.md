> 面向游戏客户端（Unity/UE）方向，涵盖计算机网络、操作系统、C++、图形学、Gameplay、数据结构等核心模块。

## 一、计算机网络

### 1. TCP 三次握手

**标准回答：**

三次握手的过程：

1. **SYN**：客户端发送 SYN 包（seq=x），进入 SYN_SENT 状态，告知服务端"我想建立连接"。
2. **SYN+ACK**：服务端收到后回复 SYN+ACK 包（seq=y, ack=x+1），进入 SYN_RCVD 状态。
3. **ACK**：客户端收到后回复 ACK（ack=y+1），双方进入 ESTABLISHED 状态，连接建立完成。

---

### 2. 为什么是三次而不是两次？

**标准回答：**

两次握手不够的核心原因是：**无法确认客户端的接收能力，且存在历史连接重用问题。**

- 若仅两次握手，服务端发出 SYN+ACK 后即认为连接建立，但它无法知道这个 ACK 客户端是否收到了。
- 更关键的是：如果一个旧的、延迟的 SYN 包到达服务端，服务端直接建立连接，浪费资源，且客户端可能拒绝这个"意外"连接，导致服务端单方面认为连接已建立。

**两次握手时谁出问题更大：**

主要是**服务端**出问题。服务端在第二次后即认为连接建立，开始分配资源，但客户端并不知情，服务端单方面维持了一个无效连接，浪费系统资源。

---

### 3. TCP 滑动窗口机制

**标准回答：**

滑动窗口是 TCP 实现**流量控制**的核心机制。

- 发送方维护一个发送窗口，窗口大小由接收方的 `rwnd`（接收窗口）决定。
- 发送方在窗口内可以连续发送数据，不必等每个包的 ACK，提高吞吐量。
- 接收方通过 ACK 中的窗口大小字段动态调整发送方的发送速率。
- 当窗口满时，发送方停止发送，等待 ACK 推进窗口。

**补充：** 拥塞控制（慢开始、拥塞避免、快重传、快恢复）是与流量控制配合的另一个机制，控制的是网络层面而非接收方层面的速率。

### 6. TCP 与 UDP 的区别

**标准回答：**

| 对比项 | TCP | UDP |
|--------|-----|-----|
| 连接性 | 面向连接（三次握手建立，四次挥手断开） | 无连接，直接发送 |
| 可靠性 | 可靠：确认 ACK、超时重传、有序交付 | 不可靠：无重传、可能乱序、可能丢包 |
| 传输方式 | 字节流（无消息边界） | 数据报（保留消息边界） |
| 头部大小 | 20~60 字节 | 8 字节（极小） |
| 速度 | 较慢（连接建立、拥塞控制等开销） | 极快（几乎无开销） |
| 拥塞控制 | 有（慢启动、拥塞避免） | 无 |
| 适用场景 | HTTP/HTTPS、文件传输、邮件 | 实时游戏、视频直播、DNS 查询 |

**为什么游戏常用 UDP：**
- 游戏对**延迟**极度敏感，TCP 的重传会导致"卡顿等待"，而游戏更宁愿丢一帧也不愿意等旧数据。
- 游戏可以在应用层**按需**实现可靠性（只对重要数据重传，如位置可以丢，技能释放不能丢）。
- KCP 协议是典型的在 UDP 上自定义可靠传输的方案（参见 RUDP 章节）。

**TCP 的粘包问题：**

由于 TCP 是字节流，发送方的两个消息可能被合并成一个包接收（粘包），或一个消息被分成多个包（拆包）。UDP 不存在此问题（每个数据报是独立的）。

解决 TCP 粘包：
- 固定长度消息头包含消息体长度，接收方按长度读取。
- 消息间加分隔符（如 HTTP 的 `\r\n\r\n`）。

### 4. 如何实现一个轻量的可靠 UDP（RUDP）

**标准回答：**

UDP 本身不可靠，要在其基础上实现可靠传输，核心思路是借鉴 TCP 的机制，在应用层实现：

1. **序列号（Sequence Number）**：每个包附带序号，接收方检测乱序/丢包。
2. **确认与重传（ACK + Retransmission）**：发送方对未确认的包启动超时重传（可用 RTO）。
3. **滑动窗口**：批量发送，避免停等协议效率低下。
4. **去重处理**：接收方对已收到的序号去重，避免重复处理。
5. **心跳/连接保活**：检测对端是否在线。

**游戏中的实际应用：** KCP 协议是常见的轻量 RUDP 实现，牺牲少量带宽换取低延迟，广泛用于实时游戏。

### 5. 帧同步 vs 状态同步

**标准回答：**

| 对比项 | 帧同步 | 状态同步 |
|--------|--------|----------|
| 同步内容 | 同步**操作（输入）** | 同步**状态（结果）** |
| 计算位置 | 客户端本地计算 | 服务端权威计算 |
| 网络带宽 | 低（只传输入） | 高（传输状态数据） |
| 反外挂 | 弱（客户端可篡改逻辑） | 强（服务端权威） |
| 延迟表现 | 所有玩家同步延迟，体验一致 | 可做客户端预测，但可能回滚 |
| 适用场景 | RTS、格斗游戏，MOBA | 三消休闲，RPG，射击，赛车游戏 |
| 确定性要求 | 高（浮点精度需统一）腾讯使用定点数，32位整数表示浮点数 | 低 |

**帧同步实现关键点：**

1. **确定性（Determinism）**：所有客户端执行相同逻辑必须得到完全相同结果。
   - 禁用浮点数（IEEE 754 不同平台结果可能不同），改用定点数（FixedPoint）。
   - 禁用随机数，用伪随机数且所有端使用相同种子。
   - 物理引擎需保证确定性（Unity 的 PhysX 不保证，需自己实现或用第三方确定性物理）。

2. **Lockstep（锁步）协议**：服务器收集所有玩家第 N 帧的输入后，统一广播，所有客户端再推进到第 N+1 帧。
   - 优点：实现简单，所有端逻辑完全一致。
   - 缺点：最慢的玩家拖慢所有人，高延迟场景体验差。

3. **乐观帧同步（Optimistic Lockstep）**：不等所有输入，客户端预测缺失输入（假设对方输入和上帧一样），服务端广播后再校验/回滚。

4. **追帧（Fast-forward）**：断线重连后快速模拟所有历史帧以追上当前进度。

**状态同步实现关键点：**

1. **服务端权威**：服务端运行完整逻辑，客户端收到状态快照后更新表现。
2. **客户端预测（Client-Side Prediction）**：客户端提前本地模拟自己的操作，无需等待服务端回包即可看到即时响应。
3. **插值/外推（Interpolation/Extrapolation）**：对其他玩家的位置做平滑插值，避免跳跃。
4. **状态回滚（Rollback）**：若服务端校验发现预测错误，回退到正确状态并重播后续输入（Rollback Netcode，格斗游戏常用）。

**实际项目参考：** 王者荣耀使用帧同步；《英雄联盟》早期帧同步，后转状态同步；FPS 类游戏基本全用状态同步+客户端预测。

## 计算机系统（操作系统）

### 1 & 11. 进程与线程的区别

一个进程至少包含一个线程（主线程），使用工厂-车间-工人的比喻来解释。

**标准回答：**

| 对比项 | 进程 | 线程 |
|--------|------|------|
| 定义 | 资源分配的基本单位 | CPU 调度的基本单位 |
| 内存 | 独立地址空间，不共享 | 共享所属进程的地址空间 |
| 开销 | 创建/切换开销大 | 创建/切换开销小 |
| 通信 | 需 IPC（管道、共享内存等） | 直接读写共享内存（需同步），线程之间通信非常快 |
| 崩溃影响 | 不影响其他进程 | 一个线程崩溃可能导致整个进程崩溃 |

### 单核CPU 能运行多线程吗？

**可以。** 单核 CPU 就像一个动作极快的厨师，他先切一刀菜（线程 1），再翻一下锅（线程 2），再放点盐（线程 3）。因为切换速度极快（微秒级），人类感觉就像是“同时”在进行。这叫**并发（Concurrency）**。而多核 CPU 才是真正的**并行（Parallelism）**。

---

### 2. 线程同步（锁、条件变量、原子操作）

**标准回答：**

锁是为了防止多个线程同时访问同一个资源，可能产生的数据混乱，而产生的机制。

- **互斥锁（Mutex）**：同一时刻只有一个线程可以持有锁，其他线程阻塞等待。适合保护临界区。
- **读写锁（RWLock）**：多个线程可同时读，写时独占。适合读多写少场景。
- **条件变量（Condition Variable）**：与锁配合使用，允许线程在某条件不满足时挂起，条件满足时被唤醒，避免忙等。
- **原子操作（Atomic）**：对单一变量的操作保证不被打断，无需加锁，性能高。适合计数器、标志位。
- **信号量（Semaphore）**：控制对有限资源的访问数量（互斥锁是信号量=1的特例）。

---

### 2.1 条件变量的实现原理

**标准回答：**

条件变量（Condition Variable）内部维护一个**等待队列**，与互斥锁配合实现"等待某条件成立再继续"。

**核心操作：**

```cpp
std::mutex mtx;
std::condition_variable cv;
std::queue<int> taskQueue;

// 消费者线程
void Consumer() {
    std::unique_lock<std::mutex> lock(mtx);
    // wait 内部：① 释放 lock ② 将当前线程加入等待队列并挂起
    // 被唤醒后：③ 重新获取 lock ④ 检查条件（防止虚假唤醒）
    cv.wait(lock, [] { return !taskQueue.empty(); });
    // 等价于：while (taskQueue.empty()) cv.wait(lock);
    Process(taskQueue.front());
    taskQueue.pop();
}

// 生产者线程
void Producer(int task) {
    {
        std::lock_guard<std::mutex> lock(mtx);
        taskQueue.push(task);
    }
    cv.notify_one();  // 从等待队列唤醒一个线程
    // cv.notify_all(); // 唤醒所有等待线程
}
```

**`wait` 的原子性保证：**

`wait` 内部的"释放锁 + 加入等待队列 + 挂起"是**原子操作**（操作系统层面保证），防止 notify 在 wait 挂起前先到达导致信号丢失。

**虚假唤醒（Spurious Wakeup）：**

操作系统可能无缘无故地唤醒等待线程（POSIX 规范允许），因此**必须用 `while` 循环（或 lambda 谓词）重新检查条件**，而不能用 `if`：

```cpp
// ❌ 错误写法（if，虚假唤醒后不再检查条件）
if (taskQueue.empty()) cv.wait(lock);

// ✅ 正确写法（while 或 lambda，确保条件真正满足）
while (taskQueue.empty()) cv.wait(lock);
cv.wait(lock, [] { return !taskQueue.empty(); });  // 等价
```

### 3. 死锁

**标准回答：**

**死锁定义：** 多个线程/进程相互等待对方持有的资源，所有线程都陷入永久阻塞。

**死锁四个必要条件（Coffman 条件）：**

1. **互斥**：资源同一时刻只能被一个线程占用。
2. **占有并等待**：线程持有资源的同时，等待获取其他线程持有的资源。
3. **不可抢占**：资源只能由占有者主动释放。
4. **循环等待**：存在线程的循环等待链（A 等 B，B 等 C，C 等 A）。

**解决方法：**

- **预防**：破坏四个条件之一，如要求一次性申请所有资源（破坏占有并等待）。
- **避免**：银行家算法，动态判断分配是否安全。
- **检测与恢复**：运行时检测死锁，强制释放资源或杀死进程。
- **实践**：锁的获取顺序全局统一；使用 `try_lock` 加超时；减小锁粒度。

### 5. Cache 是什么，实际程序中如何利用

Cache 是 CPU 与主内存之间的高速缓冲存储器，分为 L1/L2/L3 三级，速度递减、容量递增。

缓存 (Cache) vs 内存 (RAM)

- **内存 (RAM)**：你的 16GB 或 32GB 内存条。它的速度比硬盘快得多，但对 CPU 来说还是太慢了。
- **缓存 (Cache)**：集成在 CPU 内部的 L1/L2/L3 缓存。
  - **速度**：缓存 > 内存。
  - **容量**：内存 > 缓存。
  - **关系**：缓存里存放的是内存中**最常用数据的副本**。CPU 会先去缓存找数据，找不到（Cache Miss）才会去内存里搬。

因此可以让需要快速访问的数据尽量被存到缓存里，来让程序变得更快。

**利用 Cache 的核心原则：局部性**

- **时间局部性**：刚访问过的数据很快会再次访问 → 避免频繁切换数据集。
- **空间局部性**：访问某地址后，其附近地址也会被访问 → 尽量顺序访问内存。

**实际优化方式：**

- 用数组代替链表（连续内存，Cache 命中率高）。
- 遍历二维数组按行优先（row-major）。
- 热点数据放在结构体前部（Cache Line 对齐）。
- ECS（Entity-Component-System）架构将同类型组件连续存储，提升 Cache 命中。

注意：堆和栈的数据都是存在RAM，有机率命中 Cache。

### 12. 协程与线程的区别和优势

**标准回答：**

- **协程（Coroutine）**：用户态调度，一个线程内可以运行多个协程，切换不需要内核介入，开销极小。本质是函数调用，协程的切换是协作的，程序员可以通过 await 或 yield 来决定协程的切换。
- **线程**：操作系统调度，切换涉及上下文保存、内核态切换，开销较大。不同线程的切换是抢占式的，打断。

使用工厂-车间-工人-工作清单的比喻来解释。协程是为了更高效的使用线程，协程必须依附于线程，一个线程可以跑若干个协程。

**协程优势：**

1. 切换开销极小（无系统调用）。
2. 无需加锁（单线程内顺序执行，无竞争）。
3. 非常适合 I/O 密集型任务（等待期间切换到其他协程），如处理网络请求或读写文件（卡皮巴拉的协作技能系统）
4. 代码结构接近同步，可读性好（相比回调）。

**游戏中应用：** Unity 的 `IEnumerator`/`yield return` 就是协程，用于分帧执行逻辑、等待异步操作等。

Unity 引擎的工作方式就像一个极其严格的排队系统。在每一帧里，主线程会依次执行以下操作：

1. **处理输入**（Input）
2. **执行`FixedUpdate`**（物理计算）
3. **执行`Update`**（你写的逻辑）
4. **执行动画/后期处理**
5. **渲染绘制**（Rendering）

如果你有 1000 个脚本，它们的 `Update` 函数会一个接一个地在主线程上跑。如果有其中一个脚本的 `Update` 算得太慢，整个游戏就会掉帧，因为主线程被卡住了。

### 16. 堆与栈的区别

**标准回答：**

| 对比项 | 栈（Stack） | 堆（Heap） |
|--------|------------|-----------|
| 分配方式 | 自动（函数调用时） | 手动（new/malloc） |

| 生命周期 | 函数返回时自动释放 | 需手动释放（delete/free） |

| 存储内容 | 局部变量、函数参数、返回地址 | 动态分配的对象 |

### 18. 物理内存与虚拟内存

**标准回答：**

进程访问的不是真实的物理地址，而是**虚拟地址（Virtual Address）**，由操作系统的内存管理单元（MMU）通过**页表（Page Table）**映射到物理地址。

**分页机制：**

- 虚拟地址空间和物理内存都被划分为固定大小的**页（Page）**，通常 **4 KB**。
- MMU 将虚拟页号（VPN）查页表得到物理页帧号（PFN），再加上页内偏移，得到物理地址。
- **TLB（Translation Lookaside Buffer）**：页表缓存，避免每次访存都查页表（MMU 中的高速缓存）。

**缺页中断（Page Fault）：**

访问一个虚拟地址但对应物理页**不在内存中**（可能被换出到磁盘 / 从未分配）时，触发缺页中断：
1. OS 选择一个物理页帧（若已满则换出一页到磁盘）
2. 从磁盘将数据加载到该帧
3. 更新页表，重新执行触发中断的指令

**虚拟内存的好处：**
- **进程隔离**：每个进程有独立的虚拟地址空间，互不干扰。
- **超分配**：程序使用的虚拟内存可以大于实际物理 RAM（用磁盘作为补充）。
- **内存保护**：页表可标记只读、不可执行等权限，防止非法访问。
- **共享内存**：多个进程的虚拟页可以映射到同一物理页（如共享库 `.so`）。

**虚拟地址空间大小：**
- 32 位系统：4 GB（用户态通常 2~3 GB，内核态 1~2 GB）
- 64 位系统：理论 16 EB，实际 OS 限制用户态约 128 TB（Linux x86-64）

### 20. C++ 程序的内存分区

**标准回答：**

一个 C++ 进程的地址空间（虚拟内存）从低到高通常分为以下区域：

```
高地址
┌─────────────────────┐
│   内核空间（Kernel） │  操作系统使用，用户态不可直接访问
├─────────────────────┤
│   栈区（Stack）      │  ↓ 向低地址增长
│   函数栈帧、局部变量 │  自动管理，函数返回时释放
├─────────────────────┤
│   ...               │  栈与堆之间的空闲区
├─────────────────────┤
│   堆区（Heap）       │  ↑ 向高地址增长
│   new/malloc 分配   │  手动管理（或 GC）
├─────────────────────┤
│   全局/静态区        │
│   .data（已初始化）  │  全局变量、static 变量（有初始值）
│   .bss（未初始化）   │  全局/static 变量（无初始值，启动时清零）
├─────────────────────┤
│   常量区（.rodata）  │  字符串字面量、const 全局变量，只读
├─────────────────────┤
│   代码区（.text）    │  可执行机器指令，只读，多进程可共享
└─────────────────────┘
低地址
```

| 区域 | 存放内容 | 生命周期 | 管理方式 |
|------|----------|----------|----------|
| 代码区 | 机器指令 | 程序运行期间 | 只读，OS 管理 |
| 常量区 | 字符串字面量、const 全局 | 程序运行期间 | 只读，OS 管理 |
| .data | 有初始值的全局/static 变量 | 程序运行期间 | 程序启动时加载 |
| .bss | 无初始值的全局/static 变量 | 程序运行期间 | 启动时清零 |
| 堆 | `new`/`malloc` 的对象 | 手动控制 | 开发者负责释放 |
| 栈 | 局部变量、函数参数、返回地址 | 函数调用期间 | 编译器自动管理 |

**常见问题：**

```cpp
int globalA = 10;          // .data（已初始化全局变量）
int globalB;               // .bss（未初始化全局变量，默认0）
const char* str = "hello"; // str 指针在 .data，"hello" 字符串在 .rodata

void func() {
    int local = 5;         // 栈
    static int s = 1;      // .data（static 局部变量）
    int* p = new int(3);   // p 在栈，*p 在堆
}
```

### 19. 线程可以访问其他线程的栈空间吗？

**标准回答：**

技术上**可以**（同进程线程共享地址空间，可以通过地址访问），但**不应该这样做**：

1. 线程的栈随函数调用动态变化，其他线程访问时数据可能已失效（dangling pointer）。
2. 存在数据竞争，需要同步保护。
3. 正确做法：通过堆上的数据（配合锁/原子操作）或消息队列进行线程间数据传递。

### 1. C++ 多态

**标准回答：**

C++ 多态分两种：

**编译时多态（静态多态）：**
- 函数重载：同名函数，不同参数类型/数量。
- 模板：编译期展开。

**运行时多态（动态多态）：**

- 通过虚函数（`virtual`）实现。
- 基类指针/引用调用虚函数时，实际执行的是派生类重写的版本。
- 底层通过**虚函数表（vtable）**实现：每个含虚函数的类有一张虚表，对象中存储虚表指针（vptr）。

### 2. C++ 智能指针

**标准回答：**

| 类型 | 所有权 | 用途 |
|------|--------|------|
| `unique_ptr` | 独占所有权 | 资源唯一拥有，不可复制，可 move |
| `shared_ptr` | 共享所有权 | 引用计数，最后一个释放时析构 |
| `weak_ptr` | 无所有权 | 观察 `shared_ptr`，避免循环引用 |

**循环引用问题：** A 持有 B 的 `shared_ptr`，B 持有 A 的 `shared_ptr`，导致引用计数永远不为 0，内存泄漏。解决：将其中一个改为 `weak_ptr`。

### 3. 析构函数 & 为什么析构函数要是虚函数

**标准回答：**

当基类指针指向派生类对象，且通过基类指针 delete 时：
- 若析构函数**不是虚函数**：只调用基类析构函数，派生类资源无法释放，**内存泄漏**。
- 若析构函数**是虚函数**：通过虚表找到正确的派生类析构函数，先调用派生类析构，再调用基类析构，资源正确释放。

**结论：** 只要类会被继承且可能通过基类指针 delete，析构函数必须是虚函数。

### 5. 指针与引用的区别

**标准回答：**

| 对比项 | 指针 | 引用 |
|--------|------|------|
| 是否可为 null | 可以 | 不可以（必须绑定有效对象） |
| 是否可重新赋值 | 可以 | 不可以（绑定后不能改变） |
| 本质 | 存储地址的变量 | 对象的别名 |
| 算术运算 | 支持 | 不支持 |
| sizeof | 指针大小（4/8字节） | 被引用对象的大小 |
| 用途 | 动态内存、可选参数 | 函数参数、返回值 |

**指针和引用各自占多少字节：**

- **指针**：固定为 **4 字节（32位系统）或 8 字节（64位系统）**，与其指向的类型无关。`sizeof(int*)` = `sizeof(double*)` = 8（64位）。
- **引用**：`sizeof(引用)` 返回的是**被引用对象的大小**，而非引用本身的大小。引用没有独立的内存地址（C++标准不要求），但编译器内部通常用指针实现引用（占 8 字节），只是对程序员不可见。

```cpp
int x = 42;
int* p = &x;
int& r = x;

sizeof(p);  // 8（64位系统，指针大小）
sizeof(r);  // 4（等同于 sizeof(int)，即被引用对象的大小）
```

### 49. 同名函数：一个接受指针，一个接受引用，是重载吗？

**标准回答：**

**是重载。** 函数重载的条件是**同名、同作用域、参数列表不同**（参数类型、数量或顺序），与返回值类型无关。指针类型（`int*`）和引用类型（`int&`）是不同的参数类型，构成重载。

```cpp
void Process(int* p)  { /* 指针版本 */ }
void Process(int& r)  { /* 引用版本 */ }

int x = 10;
Process(&x);   // ✅ 调用指针版本（传入地址）
Process(x);    // ✅ 调用引用版本（传入对象）
// 调用方式不同，不存在二义性
```

**注意**：`const` 也可以参与重载区分：`void f(int&)` 和 `void f(const int&)` 是不同重载。

### 6. 源代码到可执行文件的流程

**标准回答：**

1. **预处理（Preprocessing）**：展开宏、包含头文件（`#include`）、条件编译 → `.i` 文件。
2. **编译（Compilation）**：语法分析、语义分析，生成中间代码/汇编代码 → `.s` 文件。
3. **汇编（Assembly）**：将汇编代码转为机器码 → `.o` 目标文件。
4. **链接（Linking）**：合并多个目标文件，解析符号引用（函数、全局变量地址），生成可执行文件。

### 💡 面试高频追问

**Q1: `#include <file>` 和 `#include "file"` 的区别？**

- **`< >`**：编译器去**标准库路径**下找。
- **`" "`**：编译器先在**当前项目目录**找，找不到再去标准库路径找。

**Q2: 什么是“符号未定义”错误 (Undefined Reference)？它发生在哪个阶段？**

- **阶段**：发生在**链接阶段**。
- **原因**：编译没问题（语法正确），但链接器在把所有文件拼在一起时，发现你调用的某个函数在所有 `.o` 文件和库文件里都找不到实现。

**Q3: 声明 (Declaration) 和 定义 (Definition) 的区别？**

- **声明**：告诉编译器“这个东西叫什么，是什么类型”（发生在编译阶段，保证语法通过）。
- **定义**：为这个东西分配实际内存（发生在链接阶段，保证能找到地址）。

### 6.1 为什么头文件不能循环引用，但 .cpp 文件可以？

**标准回答：**

**根本原因在于 `#include` 的本质是文本替换。**

预处理阶段，编译器会把 `#include "A.h"` 直接替换成 A.h 文件的全部内容。如果 A.h 和 B.h 互相 `#include`，就会触发无限展开：

```
展开 A.h → 里面有 #include "B.h" → 展开 B.h → 里面有 #include "A.h" → 无限...
```

即使有 `#pragma once` 或 include guard（`#ifndef HEADER_H`），也只能防止**同一个文件被 include 两次**，并不能解决两个类**定义上互相依赖**的问题：

```cpp
// A.h → 需要 B 的完整定义才能声明成员
// B.h → 需要 A 的完整定义才能声明成员
// 无论先编译哪个，另一个的定义永远还没出现 → 编译错误
```

**.cpp 文件为什么可以？**

`.cpp` 文件是**独立编译单元**，不会被其他文件 `#include`。`a.cpp` 调用 `b.cpp` 中的函数，只需要在编译时见到函数的**声明**（通常来自头文件），函数的具体实现在**链接阶段**才由链接器将两个 `.o` 文件拼在一起解析。所以 `a.cpp` 和 `b.cpp` 可以互相调用，只要链接时都提供了实现即可，不存在展开嵌套的问题。

**解决头文件循环依赖——前向声明（Forward Declaration）：**

当两个类需要互相引用时，用前向声明代替 `#include`，告诉编译器"这个类存在"，但暂时不需要其完整定义：

```cpp
// A.h
#pragma once
class B;          // 前向声明，不 include B.h
class A {
    B* b;         // ✅ 指针/引用只需要前向声明（大小固定为 8 字节）
    // B b;       // ❌ 值成员需要完整定义（编译器要知道 B 的大小）
};

// B.h
#pragma once
class A;          // 前向声明
class B {
    A* a;
};
```

**规则总结：**
- 使用**指针或引用**指向某类型 → 只需前向声明
- 使用**值成员、调用方法、访问字段** → 必须 `#include` 完整头文件
- 头文件之间尽量减少 include，接口尽量用指针/引用暴露，实现细节放 `.cpp`

### 7. Lambda 表达式

**标准回答：**

Lambda 是 C++11 引入的匿名函数，语法：`[捕获列表](参数列表) -> 返回类型 { 函数体 }`

**捕获方式：**
- `[=]`：值捕获所有外部变量（拷贝，只读）。
- `[&]`：引用捕获所有外部变量。
- `[x, &y]`：混合捕获。

**注意事项：**
- 引用捕获时，若 lambda 生命周期长于捕获对象，会产生悬空引用。
- `mutable` 关键字允许修改值捕获的副本。
- Lambda 底层是编译器生成的匿名类（函数对象），捕获的变量是类的成员。

### 8 继承的内存布局（含/不含虚函数）& 菱形继承

**标准回答：**

**无虚函数继承：**
```
Base: [Base成员]
Derived: [Base成员 | Derived成员]
```

**有虚函数继承：**
```
Base: [vptr | Base成员]
Derived: [vptr | Base成员 | Derived成员]  ← vptr指向Derived的虚表
```

**菱形继承问题：**
```
      A
     / \
    B   C
     \ /
      D
```
D 中包含两份 A 的成员，访问时二义性。

**虚继承解决：** `class B : virtual public A`，使 A 的数据在 D 中只保留一份。内存中 B/C 通过一个**虚基类指针（vbptr）**找到共享的 A 的数据区。

菱形继承发生在多重继承中，导致基类数据有多份拷贝，产生访问歧义。C++用**虚继承**解决，通过vbptr在运行时定位唯一的虚基类实例，代价是多一次间接寻址。虚基类的构造由最派生类负责，中间类的构造调用被跳过。实际工程中菱形继承通常是设计问题的信号，优先考虑组合或纯虚接口替代。

### 9. 虚函数 虚函数表 虚函数指针

`虚函数表 vtable    → 每个类一张，存函数指针数组`
`虚函数指针 vptr    → 每个对象一个，指向自己类的vtable`
`虚函数调用         → 通过vptr找vtable，再找具体函数`

如果子类重写了基类的虚函数，就会将vtable中对应的函数指针替换为重写的子类函数。因此子类的示例通过vptr查表得到的函数是子类重写过的。

**虚函数表（vtable）：**

- 每个包含虚函数的**类**（非对象）有一张虚函数表，存储该类所有虚函数的函数指针。
- 虚表在**编译期**确定、**程序加载时**生成，存储在**只读数据段（.rodata）**。

**虚函数指针（vptr）：**

- 每个**对象**（含虚函数的类的实例）内存中存一个 vptr，指向对应类的虚表。
- vptr 在**构造函数执行期间**赋值（先赋基类 vtable，再赋派生类 vtable）。
- 对象中不存储虚表本身，只存储指向虚表的指针。

**多重继承下的 vtable：**

- 每个有虚函数的基类对应一个 vptr，多重继承时对象中有多个 vptr。

**在构造函数中调用虚函数：**

- 可以调用，但**不会表现出多态**。构造期间 vptr 指向当前类的虚表，调用的是当前类的版本，非派生类版本。

**派生类重写虚函数后虚表变化：**

- 派生类虚表中，被重写的虚函数条目被替换为派生类的函数指针，其余继承自基类。

**构造函数不能是虚函数：**

- 虚函数调用依赖 vptr，而 vptr 在构造函数中才被赋值，构造之前对象不存在，无法通过虚表调用构造函数。

### 13. vector 扩容机制

**标准回答：**

当 `size == capacity` 时，`vector` 申请新的更大内存，将原数据复制（或移动）过去，释放旧内存。

- 扩容倍数：大多数实现为 **2 倍**（MSVC 为 1.5 倍），保证均摊 O(1) 的 push_back。
- 扩容是**代价高昂**的操作，建议提前 `reserve()` 预分配。
- 扩容后，之前的迭代器/指针/引用全部**失效**。

### 12. vector 与 list 的区别

**标准回答：**

| 对比项 | `vector` | `list` |
|--------|---------|--------|
| 底层结构 | 动态数组（连续内存） | 双向链表（节点分散） |
| 随机访问 | O(1) | O(n) |
| 头/中间插入删除 | O(n) | O(1)（已知位置） |
| 尾部插入 | 均摊 O(1) | O(1) |
| Cache 友好性 | 高（连续内存） | 低（节点分散） |
| 内存占用 | 紧凑 | 每个节点额外存两个指针 |

**实践：** 绝大多数情况下优先用 `vector`，Cache 友好带来的实际性能优势远超链表的理论插入优势。

### 38 & 44. 内存对齐规则

**标准回答：**

1. 结构体每个成员的起始地址必须是其自身大小的整数倍。
2. 结构体总大小必须是其最大对齐值（最大成员大小）的整数倍。
3. `#pragma pack(n)` 或 `__attribute__((packed))` 可强制修改对齐。

**目的：** CPU 按对齐方式访问内存效率最高，非对齐访问在某些架构上会触发异常或效率降低。

### 47. i++ 与 ++i：左值还是右值？

**标准回答：**

- `++i`（前缀）：先自增，返回自增后的对象本身，是**左值**，可以取地址。
- `i++`（后缀）：返回自增前的**临时副本**，是**右值**，不可取地址。

**性能：** 对于自定义类型，`i++` 需要构造临时对象，`++i` 更高效。

### 48. C++ 四种 cast

**标准回答：**

| cast | 用途 |
|------|------|
| `static_cast` | 编译期类型转换（基本类型、继承体系中向上/向下转，无运行时检查） |
| `dynamic_cast` | 运行时安全向下转型，失败返回 nullptr（指针）或抛异常（引用），需要 RTTI |
| `const_cast` | 去除/添加 `const` 或 `volatile` 属性 |
| `reinterpret_cast` | 底层重解释（指针与整数互转、不相关指针类型），危险，慎用 |

### 50. 左值与右值

**标准回答：**

- **左值（lvalue）**：有名字、有持久存储地址、可以取地址（`&`）的表达式。如：变量名、解引用的指针、返回引用的函数调用。
- **右值（rvalue）**：临时的、无法取地址的表达式。如：字面量（`42`、`3.14`）、算术表达式（`a+b`）、函数返回的临时对象。

```cpp
int a = 10;      // a 是左值，10 是右值
int b = a + 1;   // a+1 是右值
int* p = &a;     // ✅ 可以取左值地址
// int* q = &(a+1); // ❌ 不能取右值地址
```

**左值引用 vs 右值引用（C++11）：**

```cpp
int x = 5;

int& lref = x;          // 左值引用：只能绑定左值
// int& lref2 = 5;      // ❌ 错误，字面量是右值

int&& rref = 10;        // 右值引用：只能绑定右值（C++11）
// int&& rref2 = x;     // ❌ 错误，x 是左值

const int& cref = 10;   // 例外：const 左值引用可以绑定右值（延长临时对象生命周期）
```

**`i++` vs `++i` 再回顾：**
- `++i`：返回 i 自增后的本身 → **左值**（可取地址，`&(++i)` 合法）
- `i++`：返回自增前的临时副本 → **右值**（不可取地址）

### 51. 移动语义 & std::move

**标准回答：**

**背景问题：** 深拷贝对大对象代价高昂。例如 `vector` 扩容时复制所有元素，或函数返回一个大 `string` 时产生临时对象并立刻销毁，浪费性能。

**移动语义（C++11）：** 用**"转移资源所有权"**代替"复制数据"，将源对象的内部指针直接窃取，再把源对象置为空状态。

```cpp
class MyVector {
    int* data;
    size_t size;
public:
    // 拷贝构造：分配新内存 + 复制数据，O(n)
    MyVector(const MyVector& other) {
        data = new int[other.size];
        memcpy(data, other.data, other.size * sizeof(int));
    }

    // 移动构造：直接窃取指针，O(1)
    MyVector(MyVector&& other) noexcept {
        data = other.data;   // 窃取指针
        size = other.size;
        other.data = nullptr; // 原对象置空（防止析构时 double free）
        other.size = 0;
    }
};

MyVector a = createBigVector();  // 触发移动构造（RVO/NRVO 可能直接省略）
MyVector b = std::move(a);       // 强制触发移动构造，a 之后不可用
```

**`std::move` 的本质：**

`std::move` **不移动任何数据**，它只是一个类型转换：

```cpp
// std::move 等价于：
template<typename T>
typename std::remove_reference<T>::type&&
move(T&& t) noexcept {
    return static_cast<typename std::remove_reference<T>::type&&>(t);
}
// 本质就是：static_cast<T&&>(t)
// 将左值强制转换为右值引用，使编译器选择移动构造/移动赋值而非拷贝
```

**注意：** `std::move` 之后，原对象处于"有效但未指定"状态，不应继续使用（除非重新赋值）。

### 52. 野指针

**标准回答：**

**野指针（Dangling / Wild Pointer）**：指向无效内存区域（已释放、未初始化、越界）的指针，访问野指针是**未定义行为（UB）**，可能导致崩溃、数据损坏或静默错误。

**三种来源：**

```cpp
// 1. 未初始化指针（野指针，指向随机地址）
int* p;          // ⚠️ p 的值是随机的
*p = 10;         // 崩溃或数据损坏

// 2. delete 后未置 nullptr（悬空指针）
int* q = new int(5);
delete q;
*q = 10;         // ⚠️ q 仍指向已释放的内存
// 修复：delete q; q = nullptr;

// 3. 指针越界
int arr[5];
int* r = arr + 10;  // ⚠️ 超出数组范围
*r = 1;             // 崩溃或覆盖其他数据
```

**防御方法：**
- **首选智能指针**（`unique_ptr`/`shared_ptr`），自动管理生命周期，不会有野指针问题。
- 裸指针 `delete` 后立即赋值 `nullptr`（虽然 `delete nullptr` 是安全的，但可以防止重复 delete）。
- 用 `-fsanitize=address`（AddressSanitizer）在调试阶段检测野指针访问。

### 53. static 修饰不同对象的含义

**标准回答：**

`static` 在不同上下文中含义不同，但核心围绕两件事：**延长生命周期** 和 **限制可见性**。

**① 修饰局部变量：延长生命周期**

```cpp
void Counter() {
    static int count = 0;  // 只初始化一次，存于全局/静态区
    count++;
    std::cout << count;
}
Counter(); // 1
Counter(); // 2（count 上次的值被保留）
```

- 生命周期：程序启动到结束。
- 作用域：仍然只在函数内可见。例子中的计数器的实现。
- 初始化：C++11 保证线程安全（第一次调用时初始化）。

**② 修饰全局变量/函数：限制链接性**

```cpp
// file_a.cpp
static int gVar = 10;    // 只在 file_a.cpp 内可见，不暴露给链接器
static void helper() {}  // 同上，防止多文件符号冲突
```

- 将变量/函数限制为**内部链接（internal linkage）**，其他 `.cpp` 文件无法通过 `extern` 访问。
- 现代 C++ 中更推荐用**匿名命名空间**实现同样效果。

**③ 修饰类成员变量：类级别存储，所有实例共享**

```cpp
class Player {
public:
    static int totalPlayers;  // 声明（不分配存储）
    Player() { totalPlayers++; }
};
int Player::totalPlayers = 0;  // 类外定义（C++17之前必须）

Player p1, p2;
std::cout << Player::totalPlayers;  // 2（所有实例共享同一个变量）
```

**④ 修饰类成员函数：无 this 指针，可不实例化调用**

```cpp
class MathUtils {
public:
    static int Add(int a, int b) { return a + b; }
    // 不能访问非 static 成员（没有 this 指针）
};

int result = MathUtils::Add(3, 4);  // 不需要创建对象
```

| 上下文 | 效果 |
|--------|------|
| 局部变量 | 延长生命周期到程序结束，只初始化一次 |
| 全局变量/函数 | 内部链接，限制可见范围到当前文件 |
| 类成员变量 | 所有实例共享同一份数据 |
| 类成员函数 | 无 this，不依赖实例，类名直接调用 |

### 54. 反射（Reflection）

**反射**是指程序在**运行时**查询和操作自身类型信息（类名、字段名、方法名、类型关系）的能力。正常情况下，编写代码像是**“看着说明书组装机器”**：编译器知道所有类的名字、方法和变量。 而**反射**像是给程序装了一把**“X光机”**：程序在运行的时候，可以停下来扫描自己，问：“我手里这个对象是谁？它有哪些变量？我能不能直接改它的值，哪怕它是私有的？”

**反射就是“元数据（Metadata）”的搬运工。** 在 C# 编译时，编译器会把类名、方法名、参数名等信息打包成“元数据”存在 DLL 里。反射就是在运行时去读这本“元数据字典”的技术。

**对比 C++：** C++ 默认不存这本字典（为了追求极致速度和极小体积）。所以 C++ 原生做不到像 C# 这样一行代码就拿到私有变量名，这就是为什么 C++ 的“反射”需要像虚幻引擎（UE）那样通过复杂的宏和预处理器去“人肉”生成这本字典。

#### **优点：极高的灵活性**

- **插件系统**：你可以在不修改主程序代码的情况下，动态加载一个 DLL 文件并运行里面的类。
- **自动化工具**：如 JSON 序列化工具（JsonUtility 或 Newtonsoft.Json），它们通过反射遍历你的对象属性，自动转成字符串。

#### **缺点：性能与安全**

- **性能开销**：反射就像是在黑盒子里摸索，比直接调用代码（编译器已知地址）慢得多。在 Unity 的 `Update` 里严禁使用反射。
- **安全隐患**：它打破了“私有（private）”的限制，可能导致代码逻辑变得混乱且难以维护。

---

#### C++ 手动实现反射系统

由于 C++ 缺乏原生完整反射，工程中常用**宏 + 类型注册表**手动构建：

**核心思路：** 为每个字段记录「名称 → 成员指针（或偏移量）」的映射，运行时按名称读写。

**实际用途：**
- **序列化/反序列化**：遍历所有字段自动生成 JSON/二进制存档，无需为每个类手写 `Save()`。
- **网络同步**：自动检测哪些字段发生变化并同步（"脏标记"系统）。
- **编辑器属性面板**：运行时枚举字段并在 Inspector 中显示/编辑。
- **脚本绑定**：Lua/Python 脚本按字段名访问 C++ 对象。

---

#### Unreal Engine 的反射系统

UE 是游戏引擎中反射系统最完整的代表，通过**代码生成**实现：

**UE 反射的用途：**

| 功能 | 依赖机制 |
|------|----------|
| Blueprint 调用 C++ 函数 | `UFUNCTION(BlueprintCallable)` |

| 远程过程调用（RPC） | `UFUNCTION(Server/Client/NetMulticast)` |

### Unity 中的经典反射案例：Inspector 面板

你在 Unity Inspector 面板里看到的每一个变量，背后的核心技术就是**反射**。

- **场景**：你写了一个脚本 `Player.cs`，里面有一个 `public float speed = 5.0f;`。
- **反射的作用**：Unity 编辑器本身并不知道你写了什么。当它要把你的脚本显示在界面上时，它会用反射去“扫描”你的 `Player` 类，发现：“哦！这里有个叫 `speed` 的浮点数，类型是 `float`，值是 `5.0`。”

### 56. resize 与 reserve 的区别

**标准回答：**

两者都用于 `vector`，但操作的对象不同：

| | `reserve(n)` | `resize(n)` |
|-|-------------|------------|
| 改变 | `capacity`（容量） | `size`（元素个数） |
| 元素构造 | 不构造任何元素 | 增大时值初始化新元素，减小时析构多余元素 |
| 内存 | 预分配，避免频繁扩容 | 同时改变 size 和 capacity |

```cpp
std::vector<int> v;
v.reserve(100);   // capacity=100, size=0，无元素，无法 v[0] 访问
v.resize(100);    // capacity≥100, size=100，100个元素均值初始化为0
v[50] = 42;       // ✅ 合法（size=100）

// 常见用法
v.reserve(1000);              // 预分配，避免 push_back 触发 1000 次扩容
for (int i = 0; i < 1000; i++) v.push_back(i);
```

### 57. unordered_set

**标准回答：**

`unordered_set` 是基于**哈希表**实现的集合，元素唯一，无序。

**核心操作时间复杂度：**

| 操作 | 平均 | 最坏 |
|------|------|------|
| 插入 `insert` | O(1) | O(n)（全部哈希冲突时） |
| 查找 `count/find` | O(1) | O(n) |
| 删除 `erase` | O(1) | O(n) |

**与 set 的区别：**

| 对比项 | `set` | `unordered_set` |
|--------|-------|----------------|
| 底层 | 红黑树 | 哈希表 |
| 有序性 | 有序 | 无序 |
| 操作复杂度 | O(log n) | 均摊 O(1) |
| key 要求 | 可比较（`<`） | 可哈希（`std::hash<T>`） |

**使用注意：**
- 自定义类型作为 key 需提供 `std::hash` 特化或自定义哈希函数。
- 最坏 O(n) 可被哈希碰撞攻击利用，安全敏感场景可用随机种子哈希。

### 58. 哈希冲突 & 开放定址法的删除问题

**标准回答：**

**两种主要冲突解决方案：**

**① 链地址法（Chaining）**：每个桶存一个链表，冲突元素挂在链表上。C++ STL 的 `unordered_map/set` 采用此方式。删除直接从链表移除节点，无特殊问题。

**② 开放定址法（Open Addressing）**：冲突时在数组中找下一个空位（线性探测/二次探测/双重哈希），所有元素都存在同一数组中，Cache 命中率更高。

**开放定址法删除的问题：**

不能直接将槽位**置空（NULL）**，因为会**断开探测链**：

```
插入 A（哈希到槽 3）
插入 B（哈希到槽 3，冲突，存入槽 4）
插入 C（哈希到槽 3，冲突，存入槽 5）

删除 B → 将槽 4 置空
查找 C → 哈希到槽 3，槽 3 有 A，探测槽 4，发现空 → 误以为 C 不存在！❌
```

**解决：使用墓碑标记（Tombstone）**

删除时将槽位标记为 **DELETED**（已删除但曾被占用），而非直接置空：

- **查找时**：遇到 DELETED 继续向后探测（不停止）。
- **插入时**：遇到 DELETED 可以复用该槽位。
- **缺点**：DELETED 槽积累过多会降低性能，需要定期 rehash 清理。

### 59. 多继承：两个父类有相同的虚函数，子类重写的是哪个？

**标准回答：**

```cpp
struct Base1 { virtual void Func() { cout << "Base1"; } };
struct Base2 { virtual void Func() { cout << "Base2"; } };

struct Child : Base1, Base2 {
    void Func() override { cout << "Child"; }  // 同时覆盖 Base1::Func 和 Base2::Func
};
```

**子类重写后**：通过任何指针类型调用都执行子类版本：

```cpp
Child c;
c.Func();                              // "Child"
static_cast<Base1*>(&c)->Func();       // "Child"（Base1 的 vtable 槽指向 Child::Func）
static_cast<Base2*>(&c)->Func();       // "Child"（Base2 的 vtable 槽也被替换）
```

**子类不重写时**：调用有二义性，编译错误：

```cpp
struct Child2 : Base1, Base2 {};
Child2 c2;
c2.Func();  // ❌ 编译错误：ambiguous，需要显式指定
c2.Base1::Func();  // ✅ "Base1"
c2.Base2::Func();  // ✅ "Base2"
```

**内存布局**：多继承时对象中有两个 vptr，分别指向 Base1 子对象的 vtable 和 Base2 子对象的 vtable，两张 vtable 中 `Func` 的槽都被替换为 `Child::Func`（Base2 的版本通过 thunk 调整 this 指针偏移）。

## Unity

### 0. Unity 生命周期函数

**标准回答：**

MonoBehaviour 的生命周期按照固定顺序在主线程执行：

```
[对象创建/激活]
     │
  Awake()          ← 最先执行，哪怕对象是 inactive；做自身数据初始化
     │
  OnEnable()       ← 每次对象被激活时执行
     │
  Start()          ← 第一帧 Update 前执行一次；此时其他对象的 Awake 已完成
     │
  ┌──────────────────────────────┐  （每帧循环）
  │  FixedUpdate()  ← 固定时间步（默认 0.02s），与帧率无关；物理计算在此   │
  │  Update()       ← 每帧，处理输入、游戏逻辑                            │
  │  LateUpdate()   ← 所有 Update 结束后执行；相机跟随、IK 后处理          │
  └──────────────────────────────┘
     │
  OnDisable()      ← 对象被禁用时
     │
  OnDestroy()      ← 对象销毁时，做资源释放、取消事件订阅
```

**三个 Update 的区别与使用场景：**

| 函数 | 调用频率 | 典型用途 |
|------|----------|----------|
| `FixedUpdate` | 固定步长（默认 50 次/秒），与帧率解耦 | 给 Rigidbody 施力、物理查询（`Physics.Raycast`） |
| `Update` | 每帧一次，帧率相关 | 输入检测、角色移动（非物理）、游戏逻辑 |
| `LateUpdate` | 每帧所有 `Update` 完成后 | 相机跟随（确保角色已移动）、动画 IK 后处理 |

**Awake vs Start 的关键区别：**

- `Awake`：对象实例化时立即执行，**不依赖对象是否 active**。适合初始化自身字段、获取 `GetComponent`。
- `Start`：第一帧前执行，此时场景中**所有对象的 Awake 均已完成**。适合需要引用其他对象的初始化（避免 Awake 执行顺序不确定导致拿到 null）。

**OnEnable / OnDisable 的使用：**

在 `OnEnable` 中订阅事件，在 `OnDisable` 中取消订阅，是防止事件野引用的标准写法：

```csharp
void OnEnable()  => EventManager.OnPlayerDie += HandleDeath;
void OnDisable() => EventManager.OnPlayerDie -= HandleDeath;
```

### 4. Unity 委托（Delegate）与事件（Event）

**标准回答：**

**委托（Delegate）：**
- C# 中的**函数指针**，封装了对一个或多个方法的引用，可以像变量一样传递和调用。
- 支持**多播（Multicast）**：用 `+=` 可以注册多个方法，调用时依次执行所有已注册方法。

```csharp
// 定义委托类型
public delegate void OnPlayerDie();

// 声明委托变量
public OnPlayerDie onPlayerDie;

// 注册方法
onPlayerDie += PlayDeathAnimation;
onPlayerDie += ShowRespawnUI;

// 触发：两个方法都会被调用
onPlayerDie?.Invoke();
```

**事件（Event）：**
- 是对 Delegate 的**访问控制封装**，加了两个限制：
  1. 只有声明 event 的类（内部）才能 `Invoke()`（触发）。
  2. 外部只能 `+=` 注册或 `-=` 取消，不能直接赋值 `=` 或调用。
- 目的：强制 Observer 模式，防止外部代码随意触发或清空事件。

```csharp
public class Player : MonoBehaviour {
    public event Action OnDie;  // event 关键字

    void Die() {
        OnDie?.Invoke();  // 只有 Player 内部可以 Invoke
    }
}

public class UI : MonoBehaviour {
    void Start() {
        player.OnDie += ShowDeathScreen;  // 外部只能订阅
        // player.OnDie?.Invoke();  // ❌ 编译错误
    }
}
```

**Delegate vs Event 核心区别：**

| 对比项 | Delegate 变量 | Event |
|--------|--------------|-------|
| 外部调用 Invoke | ✅ 可以 | ❌ 不行（编译错误） |
| 外部直接赋值 `=` | ✅ 可以（危险，会清空其他订阅者） | ❌ 不行 |
| 外部订阅 `+=` | ✅ 可以 | ✅ 可以 |
| 设计意图 | 函数参数、回调 | 通知/观察者模式 |

**Unity 中常见 Action / UnityEvent：**
- `System.Action` / `System.Func<T>`：C# 内置的泛型委托，无需手动定义 delegate 类型，最常用。
- `UnityEvent`：可在 Inspector 面板拖拽赋值，常用于 UI 按钮（Button.onClick），代价是性能略低于 Action。

**实际应用：** 游戏事件总线（EventBus）、UI 刷新通知、玩家死亡通知多个系统（音效、UI、游戏管理器）。

### 5. Unity 对象池（Object Pool）

**标准回答：**

**为什么需要对象池：**
- `Instantiate()` / `Destroy()` 代价昂贵：涉及内存分配、GC 触发、组件初始化。
- 对于频繁创建销毁的对象（子弹、粒子、敌人），每帧大量 Instantiate 会导致明显卡顿。

**对象池原理：**
1. 游戏开始时**预先创建**一批对象，全部 `SetActive(false)`，放入池中。
2. 需要时从池中**取出**（SetActive true），不需要时**归还**（SetActive false），而非销毁。
3. 池满时可选：扩容（再创建新对象加入池）或等待（适合固定上限场景）。

**手动实现示例：**

```csharp
public class BulletPool : MonoBehaviour {
    [SerializeField] GameObject bulletPrefab;
    Queue<GameObject> pool = new Queue<GameObject>();

    public GameObject Get() {
        if (pool.Count > 0) {
            var obj = pool.Dequeue();
            obj.SetActive(true);
            return obj;
        }
        return Instantiate(bulletPrefab);  // 池空时扩容
    }

    public void Return(GameObject obj) {
        obj.SetActive(false);
        pool.Enqueue(obj);
    }
}
```

**Unity 内置 ObjectPool（Unity 2021+）：**

```csharp
var pool = new ObjectPool<Bullet>(
    createFunc: () => Instantiate(bulletPrefab).GetComponent<Bullet>(),
    actionOnGet: b => b.gameObject.SetActive(true),
    actionOnRelease: b => b.gameObject.SetActive(false),
    actionOnDestroy: b => Destroy(b.gameObject),
    maxSize: 50
);

Bullet b = pool.Get();
pool.Release(b);  // 归还
```

**适用场景：** 子弹、特效粒子、音效 AudioSource、敌人实例（波次刷怪）、UI 列表项（无限滚动列表）。

**注意事项：**
- 取出对象后要**重置状态**（位置、血量、速度等），否则会拿到"脏数据"。
- 配合接口 `IPoolable` / `OnSpawn()` / `OnRecycle()` 强制重置逻辑是常见设计。

**我的相关经历：**我在开发游戏 Crawler，和 chaos traffic的时候就使用了对象池。

### 6. 堆与栈在 Unity 脚本中的体现

Unity 使用 C# 编写脚本，C# 的内存模型将类型分为**值类型（Value Type）**和**引用类型（Reference Type）**，这直接对应了栈和堆的使用方式。

---

#### 值类型 vs 引用类型

| 对比项 | 值类型（Value Type） | 引用类型（Reference Type） |
|--------|---------------------|--------------------------|
| 存储位置 | 栈（或内联在其容器中） | 堆（对象本体），栈上保存引用 |
| 赋值行为 | **复制整个值**（独立副本） | **复制引用**（指向同一对象） |
| 代表类型 | `struct`、基本类型（int/float/bool）、`enum` | `class`、`interface`、数组、委托 |
| GC 压力 | 无（随栈帧自动释放） | 有（GC 负责回收） |

---

#### struct 与 class 的核心区别

```csharp
// ===== struct：值类型，赋值是复制 =====
Vector3 a = new Vector3(1, 2, 3);
Vector3 b = a;       // b 是 a 的完整副本
b.x = 99;
Debug.Log(a.x);      // 仍然是 1，a 不受影响

// ===== class：引用类型，赋值是引用 =====
public class PlayerData { public int hp = 100; }

PlayerData p1 = new PlayerData();
PlayerData p2 = p1;   // p2 和 p1 指向同一对象
p2.hp = 0;
Debug.Log(p1.hp);     // 0！p1 也被修改了
```

---

#### Unity 中常见类型的归属

| 类型 | 值/引用 | 说明 |
|------|---------|------|
| `int` / `float` / `bool` | 值类型 | 基本类型，存栈 |
| `Vector2` / `Vector3` / `Vector4` | **值类型（struct）** | Unity 内置结构体，赋值是复制 |
| `Quaternion` / `Color` / `Rect` | **值类型（struct）** | 同上 |
| `string` | **引用类型** | 虽然行为像值类型（不可变），但存堆 |
| `GameObject` / `Transform` | **引用类型（class）** | Unity 对象，存堆，通过引用操作 |
| `MonoBehaviour` 子类 | **引用类型（class）** | 脚本组件，永远在堆上 |
| `ScriptableObject` 子类 | **引用类型（class）** | 数据资产，堆上 |
| 数组 `int[]` / `List<T>` | **引用类型** | 数组本身是引用，元素若是值类型则内联 |

---

#### 重点：MonoBehaviour 一定在堆上

`MonoBehaviour` 是 `class`，不能用 `new` 创建（必须用 `AddComponent` 或 Prefab），其实例始终由 Unity 引擎管理在托管堆上。

```csharp
// ❌ 错误：MonoBehaviour 不能直接 new
var p = new Player();  // 编译警告，行为异常

// ✅ 正确：通过 Unity 生命周期创建
var go = new GameObject();
var p = go.AddComponent<Player>();
```

---

#### 装箱与拆箱（Boxing / Unboxing）——GC 的隐藏杀手

**装箱（Boxing）**：将值类型隐式转换为 `object` 或接口类型，此时值被复制到堆上，产生 GC 分配。

```csharp
int score = 100;
object boxed = score;         // ⚠️ 装箱！在堆上分配了一个 int 包装对象

// 常见隐式装箱陷阱
string s = "Score: " + score;     // ⚠️ score 被装箱（老写法）
string s2 = $"Score: {score}";    // ✅ 插值字符串，无装箱（C# 6+）

// 使用非泛型集合导致装箱
ArrayList list = new ArrayList();
list.Add(score);                  // ⚠️ 装箱

// 使用泛型集合，避免装箱
List<int> scores = new List<int>();
scores.Add(score);                // ✅ 无装箱
```

---

#### struct 在游戏性能优化中的应用

由于 struct 是值类型，不产生 GC 压力，适合高频创建销毁的数据载体：

```csharp
// 用 struct 封装每帧大量产生的数据（无 GC）
public struct DamageInfo {
    public int amount;
    public DamageType type;
    public Vector3 hitPoint;
}

// 函数返回 struct：栈上操作，零 GC
DamageInfo CalcDamage(int baseAtk) {
    return new DamageInfo { amount = baseAtk * 2, type = DamageType.Fire };
}
```

**注意 struct 的陷阱：**

```csharp
// 陷阱1：struct 作为接口使用时仍然装箱
IComparable c = score;            // ⚠️ 装箱

// 陷阱2：大 struct 的复制开销可能比引用更大
// 建议：struct 超过 16 字节时，用 ref 参数传递避免复制
void ProcessHit(ref DamageInfo info) { /* 修改不产生副本 */ }

// 陷阱3：struct 中包含引用类型字段，引用字段本身仍在堆上
public struct WeaponConfig {
    public int damage;
    public string weaponName;  // ⚠️ string 是引用类型，指向堆上的字符串对象
}
```

---

#### Unity 中减少堆分配的实践建议

| 场景 | 避免 GC 的做法 |
|------|---------------|
| 高频临时数据 | 用 `struct` 代替 `class` |
| 频繁创建的游戏对象 | 对象池（避免 `Instantiate`/`Destroy`） |
| 字符串拼接 | 用 `StringBuilder` 或 `$""` 代替 `+` |
| 集合 | 用 `List<T>`/`Dictionary<K,V>` 等泛型集合，预分配容量（`Capacity`） |
| 避免 foreach 装箱 | `List<T>` 的 `foreach` 无问题，`IEnumerable` 接口遍历有装箱风险 |
| Lambda / 闭包捕获 | 闭包捕获外部变量时会生成 class 对象，热路径上缓存委托引用 |
| 协程 | `yield return null` 每次都创建 `WaitForEndOfFrame` 等对象，可提前缓存 |

```csharp
// ✅ 缓存 WaitForSeconds，避免每次 new
WaitForSeconds wait = new WaitForSeconds(1f);
IEnumerator CountDown() {
    while (true) {
        yield return wait;  // 复用，不产生 GC
        timer--;
    }
}
```

---

#### 总结：设计 Unity 数据结构的选型原则

```
需要挂载到 GameObject / 有生命周期？
  └─ 是 → 用 class（MonoBehaviour / ScriptableObject）

纯数据容器，生命周期短/高频创建？
  └─ 是，且数据量小（≤ 16 字节）→ 优先用 struct（零 GC）
  └─ 是，但数据量大或需要共享 → 用 class + 对象池

需要多态、继承？
  └─ 必须用 class（struct 不支持继承）
```

## Gameplay 编程

### 1~3. 圆形技能范围检测（静态/移动敌人）

**标准回答：**

**静态情况：** 遍历所有敌人，判断敌人位置与技能圆心距离是否 ≤ 半径（`distance² ≤ r²`，避免开方）。

**敌人在移动且分布不均匀：** 使用空间加速结构：
- **四叉树（Quadtree）**：将2D空间递归划分，只检测与技能圆重叠的子区域中的敌人，降低检测数量。
- **格子划分（Grid）**：将地图分格，只查询圆形覆盖的格子。

**所有敌人在同一个点上：** 说明所有敌人都会被命中，无需过滤。退化为：范围检测结果为全部敌人。可以利用空间结构避免重复计算，但此场景下性能无法优化，因为所有敌人确实在范围内。

### 4. 2D 平面多个小球碰撞处理

**标准回答：**

1. **暴力法**：O(n²) 两两检测，n 小时可用。
2. **空间分区（Spatial Hashing / Grid）**：将空间划分为格子，每帧只对相邻格子内的小球进行碰撞检测，将复杂度降至接近 O(n)。
3. **Sweep and Prune**：按某一轴排序，只对轴投影重叠的对象进行精确检测。
4. **碰撞响应**：检测到碰撞后，根据弹性碰撞公式计算速度分量交换（保持动量守恒）。

### 6. Agent 与 Skill 的区别（AI 系统设计）

在游戏 AI 系统中，**Agent** 和 **Skill** 是两个层级不同的概念：

**Agent（智能体）：**

- 指具有**自主决策能力的实体**，是 AI 逻辑的载体/宿主。
- 负责感知环境（视野、威胁检测）、做出决策（行为树/状态机/GOAP）、执行行动。
- Unity 中 `NavMeshAgent` 是移动层面的 Agent，更广义的 Agent 包含完整的感知-决策-执行闭环。
- 关注的是"**谁在做决策，如何决策**"。

**Skill（技能/能力）：**
- 指 Agent 可以**执行的具体行为单元**，是离散的、可组合的动作。
- 一个 Agent 可以拥有多个 Skill（移动、攻击、眩晕、治疗等）。
- Skill 本身不做决策，由 Agent 的决策系统（行为树、技能调度器）选择**何时激活哪个 Skill**。
- 关注的是"**能做什么，怎么做**"。

**关系总结：**

```
Agent（决策层）
  ├── 感知模块（视野/听觉/威胁列表）
  ├── 决策模块（行为树 / 状态机 / GOAP）
  │     └── 选择激活哪个 Skill
  └── Skill 系统（执行层）
        ├── Skill A：近战攻击
        ├── Skill B：远程投掷
        └── Skill C：撤退移动
```

**类比：**
- Agent ≈ 一个"士兵"，有自己的判断力和目标。
- Skill ≈ 士兵的"战术动作库"（匍匐/射击/投弹），士兵根据战场情况决定用哪个。

**Unity 中的实现对应：**
- Agent：带有行为树（Behavior Designer / NodeCanvas）或自定义状态机的 MonoBehaviour。
- Skill：ScriptableObject 定义的技能数据 + 执行逻辑（如 GAS 风格的 Gameplay Ability）。

**UE 中的对应：**
- Agent：Pawn + AIController + BehaviorTree。
- Skill：GameplayAbility（GAS 中的单个技能单元），由 AbilitySystemComponent 管理调度。

### 4. 红黑树原理

**标准回答：**

红黑树是一种自平衡的 BST，通过颜色标记（红/黑）和旋转保持平衡。

**五条性质：**
1. 每个节点是红色或黑色。
2. 根节点是黑色。
3. 叶子节点（NIL）是黑色。
4. 红色节点的两个子节点都是黑色（不存在连续红节点）。
5. 从任意节点到其所有后代叶节点的路径上，黑色节点数量相同（黑高相同）。

**保证：** 最长路径（交替红黑）不超过最短路径（全黑）的 2 倍，查找/插入/删除均为 O(log n)。

`std::map` 和 `std::set` 的底层实现均为红黑树。

### 6. 遍历数组和链表哪个更快？

**标准回答：**

**数组更快**，核心原因是 **Cache 局部性**：

- 数组内存连续，CPU 预取一个 Cache Line（通常 64 字节）时，会同时加载相邻元素，后续访问命中 Cache。
- 链表节点分散在堆上，每次访问 `next` 指针都可能是 Cache Miss，触发主内存访问（延迟 100+ 倍）。

**即使理论上操作复杂度相同，实际测试中遍历数组的速度往往比链表快 5~10 倍。**

## 八、手撕算法

### 1. 无重复字符的最长子串（LeetCode 3）

**思路：** 滑动窗口 + 哈希集合

```cpp
int lengthOfLongestSubstring(string s) {
    unordered_set<char> window;
    int left = 0, maxLen = 0;

    for (int right = 0; right < s.size(); right++) {
        // 若右边字符已在窗口中，收缩左边界
        while (window.count(s[right])) {
            window.erase(s[left]);
            left++;
        }
        window.insert(s[right]);
        maxLen = max(maxLen, right - left + 1);
    }
    return maxLen;
}
```

**时间复杂度：** O(n)，每个字符最多入窗/出窗各一次。
**空间复杂度：** O(min(n, charset))，窗口大小。

---

### 2. 快速排序

**思路：** 分治。选一个基准（pivot），将数组分成"小于 pivot"和"大于 pivot"两部分，递归排序。

**核心步骤：**
1. 选取 pivot（常用随机选取或三数取中，避免最坏情况）。
2. Partition：将 pivot 放到最终位置，左边均 ≤ pivot，右边均 ≥ pivot。
3. 对左右两段递归执行 1-2。

```cpp
int partition(vector<int>& arr, int left, int right) {
    // 随机 pivot，避免已排序数组退化为 O(n²)
    int pivotIdx = left + rand() % (right - left + 1);
    swap(arr[pivotIdx], arr[right]);
    int pivot = arr[right];

    int i = left - 1;
    for (int j = left; j < right; j++) {
        if (arr[j] <= pivot) {
            i++;
            swap(arr[i], arr[j]);
        }
    }
    swap(arr[i + 1], arr[right]);
    return i + 1;  // pivot 的最终位置
}

void quickSort(vector<int>& arr, int left, int right) { 
    if (left >= right) return;
    int mid = partition(arr, left, right);
    quickSort(arr, left, mid - 1);
    quickSort(arr, mid + 1, right);
}
```

**复杂度：**

| 情况 | 时间 | 空间（递归栈） |
|------|------|--------------|
| 平均 | O(n log n) | O(log n) |
| 最坏 | O(n²)（每次 pivot 选到最大/最小值） | O(n) |
| 最好 | O(n log n) | O(log n) |

**优化——三路快排（处理大量重复元素）：**

普通快排遇到大量相等元素时退化，三路快排将数组分为 `< pivot`、`== pivot`、`> pivot` 三段，等于 pivot 的部分不再递归。

```cpp
void quickSort3Way(vector<int>& arr, int l, int r) {
    if (l >= r) return;
    int pivot = arr[l + rand() % (r - l + 1)];
    int lt = l, gt = r, i = l;
    while (i <= gt) {
        if      (arr[i] < pivot)  swap(arr[lt++], arr[i++]);
        else if (arr[i] > pivot)  swap(arr[i],    arr[gt--]);
        else                      i++;
    }
    // arr[l..lt-1] < pivot, arr[lt..gt] == pivot, arr[gt+1..r] > pivot
    quickSort3Way(arr, l, lt - 1);
    quickSort3Way(arr, gt + 1, r);
}
```

**与其他排序对比：**

| 算法 | 平均 | 最坏 | 稳定性 | 原地 |
|------|------|------|--------|------|
| 快速排序 | O(n log n) | O(n²) | 不稳定 | ✅ |
| 归并排序 | O(n log n) | O(n log n) | 稳定 | ❌（需 O(n) 空间） |
| 堆排序 | O(n log n) | O(n log n) | 不稳定 | ✅ |

**实践：** `std::sort` 通常采用 Introsort（快排 + 堆排 + 插入排序的混合），在快排递归深度超过阈值时切换为堆排，保证最坏 O(n log n)。

## 软件工程

### 1. 设计模式

设计模式是软件开发中反复出现的问题的**通用解决方案**，由 GoF（Gang of Four）在 1994 年系统整理为 23 种，分为三大类。

---

#### 三大类概览

| 类别 | 关注点 | 典型模式 |
|------|--------|----------|
| **创建型（Creational）** | 对象**如何创建** | 单例、工厂方法、抽象工厂、建造者、原型 |
| **结构型（Structural）** | 对象/类**如何组合** | 适配器、装饰器、代理、外观、桥接、组合、享元 |
| **行为型（Behavioral）** | 对象间**如何交互/职责分配** | 观察者、策略、命令、状态、迭代器、责任链、模板方法 |

---

#### 创建型模式

**单例模式（Singleton）**

保证一个类只有一个实例，并提供全局访问点。

```cpp
class GameManager {
public:
    static GameManager& Instance() {
        static GameManager instance;  // C++11 起线程安全
        return instance;
    }
    GameManager(const GameManager&) = delete;
    GameManager& operator=(const GameManager&) = delete;
private:
    GameManager() {}
};
```

**适用场景：** 游戏管理器、资源管理器、音频系统、事件总线。

**注意：** 滥用单例导致全局状态难以测试，优先考虑依赖注入。

---

**工厂方法模式（Factory Method）**

定义创建对象的接口，由子类决定实例化哪个类。将"创建逻辑"与"使用逻辑"解耦。

```cpp
// 抽象产品
class Enemy { public: virtual void Attack() = 0; };

// 具体产品
class Goblin : public Enemy { void Attack() override { /* ... */ } };
class Dragon : public Enemy { void Attack() override { /* ... */ } };

// 工厂
class EnemyFactory {
public:
    static std::unique_ptr<Enemy> Create(const std::string& type) {
        if (type == "goblin") return std::make_unique<Goblin>();
        if (type == "dragon") return std::make_unique<Dragon>();
        return nullptr;
    }
};
```

**适用场景：** 根据配置/数据动态创建不同类型敌人、技能、道具。Unity 中 `Instantiate` 配合 Prefab 本质上是工厂模式。

---

#### 结构型模式

**适配器模式（Adapter）**

将一个类的接口转换成客户期望的另一个接口，解决接口不兼容问题。

```cpp
// 旧接口
class OldAudioSystem { public: void PlaySound(int id) { /* ... */ } };

// 新接口
class AudioPlayer { public: virtual void Play(const std::string& name) = 0; };

// 适配器
class AudioAdapter : public AudioPlayer {
    OldAudioSystem old;
    std::unordered_map<std::string, int> nameToId;
public:
    void Play(const std::string& name) override {
        old.PlaySound(nameToId[name]);
    }
};
```

**适用场景：** 引擎升级时兼容旧接口、接入第三方 SDK（接口不一致时包装一层）。

---

**装饰器模式（Decorator）**

动态地为对象添加额外职责，是继承的灵活替代方案（组合优于继承）。

```cpp
class Weapon { public: virtual int GetDamage() = 0; };

class Sword : public Weapon { public: int GetDamage() override { return 10; } };

// 装饰器基类
class WeaponDecorator : public Weapon {
protected: std::unique_ptr<Weapon> weapon;
public: WeaponDecorator(std::unique_ptr<Weapon> w) : weapon(std::move(w)) {}
};

// 具体装饰：附魔
class FireEnchant : public WeaponDecorator {
public:
    FireEnchant(std::unique_ptr<Weapon> w) : WeaponDecorator(std::move(w)) {}
    int GetDamage() override { return weapon->GetDamage() + 5; }  // 叠加火焰伤害
};

// 使用
auto sword = std::make_unique<FireEnchant>(std::make_unique<Sword>());
sword->GetDamage();  // 15
```

**适用场景：** 武器附魔叠加、角色 Buff/Debuff 系统、UI 组件动态扩展。

---

#### 行为型模式

**策略模式（Strategy）**

定义一系列算法，将每个算法封装起来，使它们可以相互替换，算法的变化不影响使用算法的客户。

```cpp
// 策略接口
class MovementStrategy {
public: virtual void Move(Character& c) = 0;
};

class WalkStrategy : public MovementStrategy {
public: void Move(Character& c) override { /* 步行逻辑 */ }
};

class FlyStrategy : public MovementStrategy {
public: void Move(Character& c) override { /* 飞行逻辑 */ }
};

// Context
class Character {
    std::unique_ptr<MovementStrategy> strategy;
public:
    void SetStrategy(std::unique_ptr<MovementStrategy> s) { strategy = std::move(s); }
    void Move() { strategy->Move(*this); }
};
```

**适用场景：** AI 行为切换（巡逻/追击/逃跑）、不同寻路算法热切换、角色运动模式切换（地面/水中/空中）。

**与状态模式的区别：** 策略模式中客户端主动选择策略；状态模式中状态自己决定切换，状态之间相互知晓。

---

**命令模式（Command）**

将请求封装为对象，从而支持队列化、日志记录、撤销/重做。

```cpp
class Command { public: virtual void Execute() = 0; virtual void Undo() = 0; };

class MoveCommand : public Command {
    Character& target;
    Vector3 delta, prevPos;
public:
    MoveCommand(Character& c, Vector3 d) : target(c), delta(d) {}
    void Execute() override { prevPos = target.pos; target.pos += delta; }
    void Undo()    override { target.pos = prevPos; }
};

// 命令队列（支持撤销）
std::stack<std::unique_ptr<Command>> history;
history.push(std::make_unique<MoveCommand>(player, {1,0,0}));
history.top()->Execute();
// 撤销
history.top()->Undo();
history.pop();
```

**适用场景：** 编辑器操作的撤销/重做（Ctrl+Z）、技能队列、录像回放（记录指令序列）、帧同步中的输入指令对象化。

---

**观察者模式（Observer）**

定义对象间的**一对多依赖关系**，当一个对象状态改变时，所有依赖它的对象自动收到通知并更新。

**核心角色：**
- **Subject（被观察者/发布者）**：维护观察者列表，状态变化时通知所有观察者。
- **Observer（观察者/订阅者）**：实现更新接口，接收通知后执行相应逻辑。

```cpp
// 观察者接口
class IObserver {
public: virtual void OnNotify(const std::string& event) = 0;
};

// 被观察者
class EventSystem {
    std::vector<IObserver*> observers;
public:
    void Subscribe(IObserver* obs) { observers.push_back(obs); }
    void Unsubscribe(IObserver* obs) {
        observers.erase(std::remove(observers.begin(), observers.end(), obs), observers.end());
    }
    void Notify(const std::string& event) {
        for (auto* obs : observers) obs->OnNotify(event);
    }
};

// 具体观察者
class SoundSystem : public IObserver {
public: void OnNotify(const std::string& event) override {
    if (event == "player_die") PlaySound("death.wav");
}};

class UISystem : public IObserver {
public: void OnNotify(const std::string& event) override {
    if (event == "player_die") ShowRespawnUI();
}};

// 使用
EventSystem eventSys;
SoundSystem sound;
UISystem ui;
eventSys.Subscribe(&sound);
eventSys.Subscribe(&ui);
eventSys.Notify("player_die");  // SoundSystem 和 UISystem 都收到通知
```

**在游戏引擎中的体现：**

| 引擎实现 | 对应观察者概念 |
|----------|---------------|
| Unity `event` / `Action` | Subject 用 event 通知，Observer 用 `+=` 订阅 |
| Unity `UnityEvent` | Inspector 可视化的观察者绑定 |
| UE `Delegate` / `BlueprintAssignableEvent` | UE 的事件通知系统 |
| UE `GameplayAbility` 的 `OnGameplayEffectApplied` | GAS 中的观察者回调 |

**优缺点：**
- **优点**：Subject 和 Observer 松耦合，Subject 不需要知道 Observer 的具体类型；易于添加新的 Observer。
- **缺点**：观察者过多时通知开销增大；若忘记 Unsubscribe，观察者被销毁后 Subject 仍持有悬空指针（Unity 中 `-=` 必须配套）；通知顺序不确定，调试复杂度增加。

**与 Unity Event/Delegate 对比：**

C# 的 `event` 关键字就是对观察者模式的语言级封装：
- `event` 的声明类（Subject）负责 `Invoke()`（Notify）
- 外部（Observer）通过 `+=` 订阅、`-=` 取消订阅
- 比原始观察者模式更安全（防止外部直接触发或清空订阅列表）

**典型游戏应用：**
- 玩家死亡 → 通知音效系统、UI 系统、成就系统、存档系统
- 道具拾取 → 通知背包系统、任务系统、统计系统
- 游戏事件总线（EventBus）：全局单例 Subject，各系统按事件类型订阅

## AI 大模型

### 1. skills 和 agent

在AI领域，Skills（技能）和Agents（智能体）是两个不同层级的概念，简单来说：Skills是Agent的“工具箱”，Agent是使用这些工具的“大脑”。

1. 核心区别：零件 vs 系统

\* Skills（技能）：是原子化的单一能力。它通常只负责完成一个具体的、简单的任务。比如“查询天气”、“发送邮件”、“计算数学题”。它本身没有自主意识，需要被调用才能工作。

\* Agents（智能体）：是具备自主决策能力的系统。它拥有“大脑”（大语言模型），可以理解用户意图、制定计划、调用不同的Skills来完成任务，甚至进行反思和纠错。

2. 类比理解

\* Skills 就像“螺丝刀”：它功能单一，只能拧螺丝，但非常专业。

\* Agent 就像“机器人工程师”：它知道什么时候该用螺丝刀，什么时候该用电钻，并且能指挥自己的手去完成复杂的组装任务。

3. 工作流程

当你向一个AI Agent提问时，它内部是这样运作的：

1. 理解意图：Agent分析你的问题（如“帮我订一张去北京的机票”）。
2. 规划任务：Agent意识到这需要“查询航班”和“支付”两个步骤。
3. 调用 Skills：Agent分别调用“查询航班Skill”和“支付Skill”来执行具体操作。
4. 整合结果：Agent将两个Skill的结果整合起来，给你一个完整的答复。

Skill是Agent实现复杂功能的基础单元，而Agent是协调这些单元完成复杂任务的“总指挥”。
